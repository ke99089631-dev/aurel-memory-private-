### 実装後の発見 — Base architecture 制約
- **publicnode WSS**: subscribe 成功するが pending tx ゼロ流入（30s タイムアウト後 error）
- **Base 公式 wss://mainnet.base.org**: WS 接続自体不可（5 retry exhaust）
- **QuickNode WSS**: subscribe 成功するが Base は sequencer 集中型のため pending mempool 公開無し → event ゼロ
- → **Base は構造的に公開メンプール持たない**ことが確定

### v2 pivot — `eth_subscribe("logs")` 方式に転換
- pending tx の代わりに **確定済 Aave LiquidationCall event** を購読
- Topic 0 = keccak256("LiquidationCall(...)") でフィルタ
- 確定瞬間に他 bot の清算を検知 → skip-list 即時登録

実装：
- mempool_listener.py を全面書き直し
- ファイル名・クラス名は互換性維持
- `_process_liquidation_log` で log decode → handler 呼出
- main.py の skip-list 機構はそのまま動く

## 学び（reference 候補）
- **Base L2 の mempool 制約**: 公開 RPC では pending tx 取れない。Alchemy Flashblocks 等の専用 API か直接 sequencer 接続が必要。CYPHER のような普通の bot は logs subscription で代替するのが標準解
- **`eth_subscribe("logs")` の汎用性**: 全 RPC で対応、フィルタも豊富、Phase 2a 系の機能はこちらが本筋
- **WebSocket 接続 vs HTTP polling**: Base では logs sub の安定性 > pending tx の不確実性
- **Phase 2a を諦めず pivot した判断**: 「mempool 不可」と分かった瞬間に設計を捨てて再構築。問題のせいで止まらない
